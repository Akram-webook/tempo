## IMPORTANT: prompt vs. actual implementation
The QA prompt describes a DIFFERENT design than PR #174 actually shipped. Verified against the real
code, not the prompt's assumptions:
- Prompt: Worker writes `data/feedback.json` DIRECTLY via the GitHub Contents API + a 3-try SHA
  retry loop. **Actual: the Worker forwards to the EXISTING `receive-feedback` Action via
  `workflow_dispatch`** (one write path, reuses the tested appender). There is no direct Contents
  write and no SHA loop in the Worker — the Action's `concurrency` group serializes writes instead.
  This was a deliberate, approved decision (avoids a second writer racing the Action).
- Prompt: `feedbackEndpoint` set to a live Worker URL. **Actual: a new `feedbackProxyEndpoint`,
  EMPTY by default.** Nothing is live until Akram deploys the Worker + sets it.
- Prompt assumes `src/js/i18n.js`, `test/verify-tokens.js`, `crud(feedback):` commit messages,
  `submitFeedback()/updateFeedbackStatus()/discardFeedback()` functions, `_feedbackItems`, and
  `sample-1` item ids. NONE of these exist in this repo. Attacks depending on them were mapped to
  the real equivalents.

## Summary
Independently attacked the Worker (the entire trust boundary) and the client transport. The origin
gate, op allow-list, field whitelist, secret handling, and error paths are all correct — no secret
ever reaches the bundle or an error response, and no unauthorized origin can write. Found + fixed a
confusing no-op CORS ternary and moved the origin check ahead of the preflight. The create flow is
production-safe. The update/discard BACKEND (append-feedback `applyOp` + Action inputs) is built and
thoroughly tested, but the UI to trigger update/discard through the proxy is NOT wired yet — director
triage still writes to the per-browser sessionStorage overlay. That is the next step, not a bug in
what shipped.

## What's genuinely good
- The token never exists client-side: the proxy attaches it server-side; the bundle ships empty
  endpoints. This is the correct trust boundary.
- One write path: forwarding to the existing Action (not a parallel Contents writer) avoids the
  silent-loss class the prompt worried about entirely.
- Defence in depth on immutable fields: enforced at the proxy AND the data layer.
- Error hygiene: a thrown upstream error containing a fake `ghp_` string still does not leak — the
  Worker returns a fixed generic message.
- The create failure model preserves the user's whole draft queue (never a ghost, never lost work).

## Remaining risks (non-blocking)
- **update/discard UI not wired (OBS-174-3):** triage persistence to the shared warehouse is
  backend-ready but the exec triage UI still writes the sessionStorage overlay. Recommend a
  follow-up to wire `WP.fbTriage.set()` through the proxy (op:update/discard). Small, the backend
  is done and tested.
- **Go-live is an owner step:** deploy the Worker + `wrangler secret put GITHUB_PAT` + set
  `feedbackProxyEndpoint`. Until then Submit safely stays local (no leak). CORS attacks F1-F7 were
  verified in a faithful Node simulation of the Worker; a live `curl` re-test should be run once

- GitHub Pages deploy lag (~5-10 min) before a committed item appears in the served JSON — by design.
