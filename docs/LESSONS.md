# Lessons

A running log of what worked, what nearly broke, and the rule to remember. Append newest at the top.

## 2026-07-14 — Executive Status: build the thing, don't link to it
- **What worked:** The "deck" started as a config-driven link to an external Google Slides artifact (hidden until a URL was pasted). Akram then asked to actually create it — so I built it as a live in-app page off the SAME capacity/growth/access engine the dashboard already uses. Reusing the existing engine meant the whole page was real data on day one (no mock numbers, nothing to keep in sync) and only needed a view + a gate. Placement: a nav tab under Dashboard + the existing dashboard card repointed to navigate internally — two good entry points, and I removed the now-redundant account-menu item rather than keep three.
- **What nearly broke / the gotcha:** The dashboard card was an `<a href target=_blank>` opening an external URL; repointing it to an internal route meant switching it to a `<button data-go>` — and the CSS assumed an anchor, so the button needed `font:inherit; cursor:pointer; width:100%; text-align:start` or it'd inherit UA button styling. Also had to retire the old plumbing (`execDeckUrl`/`execDeckVisible`) so no dead config lingered, and re-point the test from "opens a URL in a new tab" to "navigates internally (no href/target)".
- **What I'd do differently next time:** When a feature is first scoped as "link to an external artifact," flag early that building it in-app is often cheaper AND better when the data already lives in the app — an external deck is a maintenance burden (someone updates slides) that a live page eliminates. Prefer the self-updating version when the engine exists.
- **Rule to remember:** If the data is already in the product, build the view — don't link out to a static copy that instantly goes stale. Reuse the existing engine; a new surface should be a view + a gate, not a new data path. When you change what a control does (external link → internal nav), update its element type, its CSS assumptions, AND invert its test.

## 2026-07-14 — Trim pass 2: remove Role/Date, Workspace = Members only
- **What worked:** Akram trimmed again — cut Role + Date format from My settings and reduced Workspace to only Members & Access. When a whole block goes, I deleted its functions AND its i18n-driven UI AND its event wiring in one pass, then inverted the guarding tests to assert ABSENCE. Kept a clean seam: the activity *page* route stayed (something else may link to it) while only its Settings shortcut button was removed — removing a shortcut ≠ removing the destination.
- **What nearly broke / the gotcha:** `verify-settings-layout` was almost entirely about the tier editor + Slack list — two features now gone. Rather than delete the suite, I repurposed it to assert the Workspace tab is now Members-only (tiers/slack/roles/activity-button all absent), so it still earns its place as a regression guard for the trim. A test whose subject was removed should be re-pointed at the new invariant, not silently dropped — otherwise you lose the guard that the old thing stays gone.
- **What I'd do differently next time:** Ask up front "does the destination stay, or just the entry point?" — I nearly removed the whole activity route when only the button was meant to go. Entry-point vs destination is a distinct decision; confirm it before cutting.
- **Rule to remember:** Removing UI = delete code + strings + wiring + invert the guarding test (assert it's gone), and re-point (don't delete) any suite whose subject was removed so the removal stays enforced. Distinguish "remove the shortcut" from "remove the feature."

## 2026-07-14 — Trim My settings (Akram feedback: less is more)
- **What worked:** Akram reviewed the shipped Settings and called out a lot as noise (density, date-format picker, quiet hours, digest, in-app channel, last-sign-in, device row, sign-out-everywhere, the whole Privacy catalogue + export). Rather than defend the "complete" version, cut hard to what a person actually uses: theme+language, three notification topics × two channels, change-password, one privacy line. Removed the entire `WP.privacy` module + its test instead of leaving dead code behind a hidden feature. Kept the *plumbing* that others depend on (density default still applied, `WP.fmt.date` still formats) — only the UI controls went. Answered "what does role mean?" with a per-level one-liner instead of the raw enum.
- **What nearly broke / the gotcha:** Two shipped test suites (`verify-security`, `verify-settings-v2`) asserted the presence of the very things being removed — so the cut *had* to come with rewriting them to assert ABSENCE (`!/sec-signout-all/`, `!/data-seg="density"/`). A removal isn't done until the test that guarded the old behaviour is inverted; otherwise the suite fails and you can't tell a regression from an intended cut.
- **What I'd do differently next time:** When a feature ships "complete," expect the real signal after someone uses it — build the first version lean enough that trimming is cheap, and don't over-invest in sub-features (quiet hours, digest, export) before there's a pull for them. The audit-driven "add everything useful" instinct fights the Constitution's Simplicity article; bias to the smallest thing that answers the job.
- **Rule to remember:** Ship the lean version and let use tell you what to add — removing shipped UI costs test rewrites + a deploy, adding later is one PR. When you remove UI, invert its guarding tests (assert it's GONE) and delete any module that only existed for it. Simpler wins absent evidence (Constitution, Simplicity).

## 2026-07-14 — Settings → Privacy (what Tempo tracks about me + export)
- **What worked:** Grounded "what we track" in what the code ACTUALLY stores (SAVE_KEYS/prefs, the append-only events store with per-event `source`, the activity log, the directory record) instead of writing marketing copy — every category on screen points at a real store, so it can't drift into a lie. The no-surveillance stance is enforced by construction: the events store already throws on an event with no `source`, and the privacy catalogue is a fixed list with no keystroke/presence/location/screen category — the test asserts none can be added. Export is a client-side Blob download (no server, no new endpoint) and `buildExport(data, now)` takes the timestamp as an argument so it's testable and free of a buried `Date.now()`.
- **What nearly broke / the gotcha:** My first test "no foreign events" check was vacuous — `myData()` strips `subjectId` from each event for the owner-facing shape, so filtering the *output* by `subjectId` always found nothing regardless of correctness. Fixed by asserting against the STORE (`WP.events.query(self)`) and checking `myData` returns exactly that count — a real guard, not a tautology. Also: a data-export must gather at the SELF layer (filter before you strip identifying fields), never trust the post-shaping output to prove scoping.
- **What I'd do differently next time:** When testing a filter, assert on something the filter could actually get wrong — if the shape you assert on has already discarded the field the filter keys on, the test proves nothing. Check the boundary (store → my-data), not the already-shaped tail.
- **Rule to remember:** For any "your data" surface, enumerate real stores (not aspirational categories), keep provenance (`source`) on every row, scope to self at the query layer, and make timestamps an injected argument. A privacy claim you can't point at a store for is a claim you shouldn't render.

## 2026-07-14 — Executive status entry point (deck link, role-gated)
- **What worked:** One gate helper (`WP.execDeckVisible()` = non-empty URL AND `WP.can('viewSettings')`) drives BOTH entry points, so the card and the nav item can never drift out of sync — a member can't see one without the other, and flipping the URL empty hides both at once. Kept the deck as a config-driven link that opens in a new tab (never embedded — it's a Google-auth surface). Reused the existing `external` icon and the `.acct-item` menu row (added only an `a.acct-item` text-decoration reset).
- **What nearly broke / the gotcha:** I first put the card inside `director()`, but the super-admin (`p_akram`) has `level === 'manager'`, so they render `leader()`, not `director()` — the card vanished for the very person who most needs it. The role *label* and the dashboard *variant* are different axes: `WP.can('viewSettings')` (capability) ≠ `viewer.level === 'director'` (org level). Fixed by rendering the card at the top of `render()` for anyone who passes `execDeckVisible()`, independent of which dashboard variant drew.
- **What I'd do differently next time:** When a feature is gated by a *capability*, place it where the capability is checked — not inside a branch keyed on org level/role name. Don't assume "director dashboard" == "everyone with director-level access."
- **Rule to remember:** Capability (`WP.can`) and org level (`viewer.level`) are separate axes; gate on the one that actually defines who should see the thing, and render it outside any role-variant branch. One gate helper for multi-placement features keeps every placement consistent.

## 2026-07-14 — Settings → Security (password · devices · sign-out-everywhere)
- **What worked:** Reused B1's `resetPasswordForEmail` for "change password" (email a secure link, never handle the old password client-side) and Supabase `signOut({scope:'global'})` for sign-out-everywhere. Was honest about what we can't do yet: no per-device session list (needs the sessions service) and 2FA marked "coming soon" — placeholders, not fake controls.
- **What nearly broke / the gotcha:** (1) The Security helpers first called `sbClient()`, which in jsdom injects the Supabase CDN `<script>` that never loads → the promise hung → the test's async IIFE never finished → Node exited 0 with NO output (a silently "passing" test). Fixed by using the already-live `WP._sb` and reporting offline immediately (also a real robustness win: a settings button should never hang on a CDN). (2) Reusing the `.mbr-row` class for the "this device" row broke `verify-members`' assertion that `mbr-row` isn't on the personal tab — gave it a distinct `.sec-device` class. (3) `sign-out-all` log entry doesn't survive because sign-out resets the in-memory activityLog — asserted via a `logEvent` spy instead.
- **What I'd do differently next time:** A test that exits 0 with NO PASS line is FAILING silently — always confirm the PASS string prints, not just the exit code. And don't call a CDN-injecting helper from a UI action; use the live client or fail fast.
- **Rule to remember:** UI actions must never hang on an external load — use the already-initialized client and fall back immediately. Reusing a CSS class couples unrelated features to each other's tests; give a distinct concern a distinct class.

## 2026-07-14 — Members & Access (allowlist becomes a screen)
- **What worked:** Read the access engine FIRST and found `grantAccess/listAccess/setAccess` already existed and already persisted (via `persist.js` `granted`), plus a whole `permissions.js` role screen. So the "big gap" was only surfacing the allowlist in Settings — I built the roster + grant/revoke toggle and *linked* to the existing role screen instead of rebuilding role-change logic. Less code, no duplication.
- **What nearly broke / the gotcha:** `grantAccess()` logs but does NOT persist on its own — the caller must `WP.setState({})` or the revoke is lost on reload. Also the roster must exclude `tbc` (open/unfilled roles) — they're not real members and would show fake toggles. And self-revoke had to be blocked or an admin could lock themselves out.
- **What I'd do differently next time:** Before building any "manage X" screen, grep for an existing engine/API (`grep -n "function .*[Gg]rant\|setAccess\|listAccess"`) — the capability often already exists headless and just needs a UI. Don't assume it must be built from scratch because it "lives in code today."
- **Rule to remember:** A mutation helper that logs is not the same as one that persists. After any `WP.access.*`/data mutation from the UI, call `WP.setState({})` so `persist.saveData()` runs. Reuse + link beats rebuild.

## 2026-07-14 — Settings v2 (personal / workspace split, preferences, notifications)
- **What worked:** Researched world-class settings IA first (Nicelydone / memorable.design / Equal.design) instead of guessing — confirmed the personal-vs-workspace split, "microcopy under every control", and the notifications model (channel × category + digest + quiet-hours). Reused existing components (`.seg`, `ui.subTabs`, `overlay-host`) rather than inventing new ones, so the new UI inherited house styling for free. Added `WP.prefs` with a `mergePrefs()` deep-merge so new pref keys never come back undefined on an old saved session.
- **What nearly broke / the gotcha:** Moving admin org-config under a "Workspace" tab silently broke `verify-settings-layout.js`, which rendered Settings and looked for tier/Slack markup that is now behind a non-default tab. Fix: the test sets `WP._settingsTab = 'workspace'` before asserting. Also the `__admin__` sentinel viewer has no `email`/`byId` row — the Account section had to guard (`WP.viewer()` + optional `roleLabel`) or it would throw.
- **What I'd do differently next time:** When splitting a page into tabs, grep the tests for anything that renders that page BEFORE running the suite — you can predict the break and fix the test in the same edit.
- **Rule to remember:** A preference control must actually DO something and PERSIST + confirm — `data-density` was inert until a CSS rule consumed it. Wire the effect, the persistence, and the saved-toast together, or it's a fake setting.
